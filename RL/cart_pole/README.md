# CartPole — policy gradient: REINFORCE и PPO

Два on-policy policy-gradient подхода к среде `CartPole-v1` на PyTorch — от
базового к современному. Оба агента написаны с нуля.

- [`reinforce_agent.py`](reinforce_agent.py) — **REINFORCE** (Monte-Carlo policy gradient) с опциональным baseline
- [`ppo_agent.py`](ppo_agent.py) — **PPO** (Proximal Policy Optimization)

> `CartPole-v1` обрывает эпизод на 500 шагах, reward = число удержанных шагов.
> Поэтому reward **500 = идеальная политика**.

---

## 1. REINFORCE (`reinforce_agent.py`)

Классический Monte-Carlo policy gradient: играем **полный эпизод**, считаем
дисконтированные returns `G_t`, двигаем политику в сторону `log π(a|s) · advantage`.

- **Actor** (`PolicyNet`) — политика `π(a|s)`
- **Critic** (`ValueNet`) — baseline `V(s)`, создаётся только при `use_baseline=True`
- Advantage: `G_t − V(s)` с baseline или просто `G_t` без него

Эксперимент с baseline (бюджет 1000 эпизодов):

| `use_baseline` | Результат |
|----------------|-----------|
| **True** | стабильно доходит до **500** и решает задачу (ранняя остановка) |
| **False** | за все 1000 эпизодов упирается в **~300**, задачу не решает |

Baseline снижает дисперсию градиента — без него редкие «удачные» эпизоды тянут
веса хаотично. Но даже с baseline REINFORCE обучается **по одному эпизоду за раз**
и остаётся sample-inefficient — отсюда переход к PPO.

## 2. PPO (`ppo_agent.py`)

Современный on-policy алгоритм, устраняющий слабости REINFORCE.

- **Общая сеть** `ActorCritic` — единое тело + две головы (actor, critic)
- **Векторизованные среды** (`num_envs=8`) — параллельный сбор опыта rollout'ами по `n_steps=128`
- **GAE(λ)** — сглаженная оценка advantage, баланс между смещением и дисперсией
- **Clipped surrogate objective** — обрезка отношения вероятностей `clip(ratio, 1±ε)` не даёт политике меняться слишком резко за один шаг (ключевая идея PPO)
- **Несколько эпох по минибатчам** (`n_epochs=10`) на одном rollout — переиспользование данных → sample-efficiency
- Entropy bonus (исследование), value loss, gradient clipping

### Результаты PPO

- Средняя награда **на обучении ≈ 250** — метрика усредняет эпизоды, собранные
  *стохастической* политикой (с entropy-бонусом на исследование), поэтому она
  ниже реального качества выученной политики.
- **Eval (детерминированный, `argmax`): стабильно 500** — жадная политика идеальна.

Детерминированное vs стохастическое поведение на eval:

| Режим | Награды по эпизодам |
|-------|---------------------|
| детерминированный (`argmax`) | 500, 500, 500 — идеально |
| стохастический (сэмплирование) | 500, 83, 491 — большой разброс |

Разброс в стохастическом режиме — прямое следствие энтропии: entropy bonus
намеренно держит политику «шумной» ради исследования, поэтому при сэмплировании
изредка выбирается плохое действие и эпизод рушится. Жадный `argmax` этого шума
лишён — отсюда стабильные 500. Это же объясняет, почему train-среднее (250) сильно
ниже eval (500).

## Сравнение

| | REINFORCE | PPO |
|---|-----------|-----|
| Сеть | раздельные actor / critic | общий `ActorCritic` |
| Сбор данных | 1 эпизод, 1 среда | rollout `n_steps × num_envs`, векторно |
| Advantage | `G_t − V(s)` | GAE(λ) |
| Обновление | 1 шаг на эпизод | clipped surrogate, `n_epochs × minibatch` |
| Свойства | высокая дисперсия, sample-inefficient | стабильный, sample-efficient |
| Результат | 500 (с baseline) / ~300 (без) | eval 500 (train-avg ~250) |

## Ключевые гиперпараметры PPO

| Параметр | Значение | Смысл |
|----------|----------|-------|
| `num_envs` | 8 | параллельных сред |
| `n_steps` | 128 | длина rollout на среду |
| `gamma` | 0.99 | дисконт наград |
| `gae_lambda` | 0.95 | сглаживание GAE |
| `clip_eps` | 0.2 | ширина клиппинга политики |
| `n_epochs` | 10 | проходов по собранным данным |
| `ent_coef` | 0.01 | вес энтропии (исследование) |

## Запуск

```bash
python RL/cart_pole/reinforce_agent.py   # REINFORCE
python RL/cart_pole/ppo_agent.py         # PPO
```

Зависимости: `gymnasium[classic-control]`, `torch`, `numpy`, `matplotlib`.
У обоих агентов есть `train()`, `evaluate()`, `plot_progress()` и `watch()` —
визуализация политики в отдельном окне.
